Tool Calling

3 / 11 scored 0 — Ollama tag limitation

What it tests: 5 structured function-calling tasks. Each model must select the correct function and supply correctly-typed arguments via the Ollama /api/chat tools API. Tests whether the model tag exposes a function-calling schema at all.

Why it matters: Tool calling is the primitive that turns a chat model into an AI agent. Without it, every action — calendar lookup, file operation, API call — requires the model to describe the action in prose, which your code then has to parse. For Phase 2 of this project (email, calendar, Telegram), tool calling is a hard requirement, not a nice-to-have.

Positional Code Recall (codeneedle)

Gemma 4 26B: 82%  ·  Next best: 36%

What it tests: Given a full codebase (~5K tokens, http_server corpus), reproduce specific named functions from their position in the file. Harder than NIAH: models must retain syntactic structure, indentation, and logic — not just a fact string — across a long code context.

Why it matters: Code assistants (PR review, refactoring, explain-codebase) must recall exact implementation details. A model with 9% positional code recall will hallucinate function bodies and generate broken refactors even when the source is in its context window. This is the test that separates general recall from code-specific memory.

Coding Benchmark — HumanEval+

Results invalid — Windows harness bug

What it tests: 20 Python function completions from HumanEval+ (EvalPlus). Each solution is executed against an extended test suite with 80× more tests than base HumanEval. Measures whether a model produces syntactically and semantically correct code — not just retrieval, but generation accuracy.

Why it matters: pass@1 on HumanEval is the standard coding capability ceiling used across the industry. It answers: "if the model writes a function once, how often is it correct?" Directly relevant for any assistant used in a software engineering workflow.

⚠ All 11 models scored 0 / 20 — harness bug, not model failure

Solutions were generated and appear correct on manual inspection. The scoring step failed silently.

Root cause: evalplus.evaluate v0.2.1 calls resource inside each multiprocessing worker (evalplus/eval/utils.py:116) for sandbox memory limiting. resource is a POSIX-only module — it does not exist on Windows. All worker processes crash with ModuleNotFoundError: No module named 'resource', producing empty subprocess output. _parse_pass_at_1() then returns 0.0 on empty output, making it appear as if every model solved zero problems.

Planned fixes:

  • Patch reliability_guard to skip resource on Windows — use psutil for memory monitoring or skip memory limits entirely
  • Route the evalplus.evaluate subprocess through WSL while keeping Ollama on Windows
  • Write a custom Windows-compatible executor that runs test cases without POSIX sandbox primitives
